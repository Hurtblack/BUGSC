# 任务查询独立入口与任务展示统一设计

## 背景

当前任务数据已经独立沉淀在 `app/src/main/assets/blueprint/scmdb_missions.json`，并由 `BlueprintDataRepository.loadAllMissions()` 提供全量读取能力。

但现有产品层存在两个问题：

1. 任务浏览被隐藏在“蓝图图鉴 -> 任务”模式里，用户心智不直观。
2. 蓝图图鉴中的任务卡片信息不足，且部分任务标题、派系、地点等字段虽然有数据，但没有统一转换成稳定的显示字段，导致“有值但显示不友好”。

本次改动目标是新增独立“任务查询”入口，同时统一任务显示规则，让“任务查询页”和“蓝图图鉴内的任务展示”共享同一套显示语义。

## 目标

### 产品目标

- 在“查询”首页新增独立“任务查询”入口。
- 提供专门的任务查询页，支持：
  - 直接搜索
  - 按星系筛选
  - 按派系筛选
  - 组合筛选后浏览任务列表
- 蓝图图鉴中的任务模式继续保留，但显示效果与新任务查询页保持一致。

### 体验目标

- 用户不需要先理解蓝图系统，也能直接进入任务查询。
- 搜索既可以搜标题，也可以搜派系、星系、地点、任务类型。
- 任务卡片的信息密度高于当前实现，列表页即可判断任务是否值得点开。

## 非目标

- 本次不新增远程接口，不改动任务数据源结构。
- 本次不做任务收藏、排序自定义、历史记录。
- 本次不做公共 UI 组件大范围抽象；仅在数据/显示规则层做必要复用。
- 本次不重构 `MissionDetailSheet` 的整体结构，只复用现有详情弹窗。

## 现状分析

### 数据层

现有任务数据来源已经满足需求：

- `scmdb_missions.json` 提供全量任务详情
- `RewardMission` 已承载标题、派系、星系、地点、报酬、标签等核心字段
- `BlueprintDataRepository.loadAllMissions()` 已支持全量任务读取
- `MissionDetailSheet` 已支持按 `guid` 展示详情

因此本次无需新增仓库或数据同步链路，重点在于：

- 增强 `RewardMission` 的显示派生字段
- 在查询页和蓝图页复用同一套过滤与显示规则

### 页面层

当前“查询”首页只有蓝图、飞船、维克洛、矿物入口，没有独立任务入口。

当前任务浏览挂在 `BlueprintFragment` 的“任务”模式下，具备基础搜索能力，但仅支持：

- 标题匹配
- 英文标题匹配
- 类型匹配

当前缺失：

- 星系筛选
- 派系筛选
- 地点搜索
- 派系搜索
- 卡片副信息统一展示规则

## 方案对比

### 方案 A：继续扩展 BlueprintFragment

在蓝图图鉴内继续强化现有“任务”模式，并从查询首页直接跳到该页面。

优点：

- 改动最小
- 复用现有页面最多

缺点：

- 蓝图图鉴页面职责继续混杂
- 后续任务查询增强会继续挤压蓝图页结构
- 从信息架构上仍然不够直观

### 方案 B：新增独立 MissionQueryFragment

在查询首页新增“任务查询”入口，跳转到独立任务查询页；蓝图图鉴中的任务模式继续保留，但共享同一套显示规则。

优点：

- 入口清晰，符合用户心智
- 页面职责分离明确
- 便于后续继续增加任务筛选或排序

缺点：

- 需要新增页面与导航配置
- 任务列表逻辑需要从蓝图页现有实现中提炼出可复用部分

### 方案 C：先抽完整公共任务浏览组件，再接两个页面

优点：

- 长期结构最干净

缺点：

- 本次改动范围偏大
- 为当前需求引入过早抽象

### 结论

采用方案 B。

理由：

- 它直接解决“任务查询入口被埋”的核心问题。
- 它允许蓝图图鉴继续保留任务模式，不破坏现有用户路径。
- 它只要求在数据与显示层做适度复用，不需要本次就承担完整组件抽象成本。

## 设计

### 信息架构

在“查询”首页新增一张“任务查询”卡片，点击进入独立 `MissionQueryFragment`。

信息架构调整后：

- 查询首页
  - 蓝图图鉴
  - 任务查询
  - 飞船查找
  - 维克洛兑换
  - 矿物查询

蓝图图鉴仍保留“任务”模式，用于蓝图语境下的任务查看；任务查询页则承担通用任务检索入口。

### 任务查询页结构

页面结构采用“顶部筛选 + 直接列表”的两段式浏览方式：

1. 顶部搜索框
2. 星系筛选 chips
3. 派系筛选 chips
4. 任务列表
5. 点击卡片打开 `MissionDetailSheet`

默认状态：

- 搜索词为空
- 星系 = 全部
- 派系 = 全部
- 任务列表按默认顺序展示

默认排序：

- 先沿用当前任务模式的排序策略，以 `rewardUec` 降序为主
- 无报酬任务保持稳定尾排

这样可以避免本次同时引入新的排序争议。

### 筛选行为

筛选规则采用交集：

- 星系筛选限制任务所属星系
- 派系筛选限制任务派系
- 搜索始终在筛选结果上继续匹配

匹配规则：

- 星系：基于 `availableSystems`
- 派系：基于 `factionNameCn`，无中文时回退 `factionName`
- 搜索：匹配任务聚合搜索字段

搜索字段包含：

- 中文标题
- 英文标题
- 任务类型中英文
- 派系中英文
- 星系
- 地点名
- 目的地名

### 任务显示模型统一

为避免“页面自己拼字符串”导致结果不一致，需要把 UI 所需的任务显示语义收敛到 `RewardMission` 或其邻近 helper。

至少补齐这些派生字段：

- `displayTitle`
  - 优先中文标题
  - 回退英文标题
  - 再回退任务类型
- `displayFaction`
  - 优先中文派系名
  - 回退英文派系名
- `displaySystems`
  - 从 `availableSystems` 生成展示文本
- `displayLocationsSummary`
  - 从地点与目的地生成简短摘要
- `searchTokens`
  - 预聚合搜索文本，避免每次 filter 时重复拼装
- `cardMetaLine`
  - 用于列表卡片副信息展示

如果标题字段存在占位、空白或明显不适合作为主标题的内容，应在同一层做统一兜底，而不是在 Fragment 内写特判。

### 任务卡片展示规则

任务查询页与蓝图图鉴内任务模式统一使用相同的信息密度。

卡片展示建议：

- 第一行：任务标题
- 第二行：派系 / 星系 / 类型
- 第三行：报酬 / 标签 / 蓝图数量

标签优先展示：

- 战斗
- 非法
- 可共享
- 一次性

蓝图数量仅在存在关联蓝图时显示。

报酬优先显示为 `aUEC` 格式化值；如果无报酬，则该字段省略而不是显示空占位。

### 蓝图图鉴中的任务模式调整

`BlueprintFragment` 中当前“任务”模式保留，但需要改成与 `MissionQueryFragment` 共享同一套任务显示规则。

这意味着：

- 卡片标题规则一致
- 副信息规则一致
- 搜索字段范围一致

但蓝图图鉴内任务模式仍然可以保持更轻的交互，不要求本次把星系/派系筛选也完全搬进去。最低要求是列表显示统一、搜索能力增强、标题兜底一致。

### 详情页复用

`MissionDetailSheet` 继续按 `guid` 加载详情，不新增第二份详情实现。

这保证：

- 入口变多，但详情来源唯一
- 任务数据修复只需做一次

## 验收标准

### 功能验收

- 查询首页存在独立“任务查询”入口
- 点击后进入独立任务查询页
- 任务查询页支持：
  - 搜索
  - 星系筛选
  - 派系筛选
  - 组合筛选
- 点击任务卡片能打开现有详情弹窗
- 蓝图图鉴中的任务模式仍可正常使用

### 显示验收

- 任务标题不再大量出现空白、退回异常值或缺失主标题的问题
- 新任务查询页与蓝图图鉴中的任务卡片展示风格一致
- 搜索“派系名 / 星系名 / 地点名 / 类型名”可以命中相关任务

### 代码验收

- 任务显示规则不分散在多个 Fragment 中重复拼接
- 过滤逻辑对空字段安全
- 新页仅新增必要逻辑，不引入与现有页面职责冲突的结构

## 风险与处理

### 风险 1：派系和星系值过多，chip 过长

处理：

- 首版仅展示实际出现的值
- 保留“全部”作为默认选项
- 如果 chip 数量过多，允许横向滚动或换行布局，具体以当前项目布局风格为准

### 风险 2：标题清洗规则过度，误伤正常标题

处理：

- 先以“优先中文标题、回退英文标题、再回退类型”为主
- 仅在明显空白、占位或无意义时兜底，不做激进改写

### 风险 3：两个页面的任务列表逻辑再次分叉

处理：

- 在模型或 helper 层统一卡片字段与搜索 token
- Fragment 只负责状态与渲染，不自己重新拼展示文本

## 后续可扩展方向

- 增加排序：按报酬、按战斗/非战斗、按掉蓝图数量
- 增加任务类型 chip
- 增加“仅看蓝图任务”或“仅看可共享任务”快捷筛选
- 增加收藏与最近查看
